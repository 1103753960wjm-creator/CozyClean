# CozyClean Flutter Engineering Rules

本文件定义 Flutter 工程层规则，强调“与现有项目一致”，禁止为满足模板规则而大规模重构。

## 1. 技术基线
- 以仓库当前稳定技术栈为准。
- 新代码必须与现有风格和依赖体系兼容。
- 非必要不引入新框架，不为“规范完整性”做大迁移。

## 2. 状态管理
- 使用 Riverpod（与项目现状保持一致）。
- 状态必须不可变；列表对外暴露不可变视图。
- 禁止在业务逻辑里直接可变修改状态集合。

## 3. UI 性能与线程规则
- `build()` 必须保持纯函数。
- 重计算、IO、相册扫描、分组计算放到 controller/service/isolate。
- 长耗时流程必须可观测（日志）且可降级。

## 4. 日志与可观测性
- 使用 `debugPrint()`，禁止 `print()`。
- 日志禁止输出用户隐私和真实文件路径。
- 关键路径至少记录：开始、结束、耗时、成功/失败、核心计数（如扫描数量）。

## 5. 编码与乱码防护（MANDATORY）
- 所有规则文件、Dart 文件、文档文件统一 `UTF-8` 编码。
- 终端日志文件写入必须显式使用 UTF-8。
- 出现中文乱码时，优先检查文件编码与终端编码，再进入业务排查。

## 6. 导入与命名
- 文件命名采用 `snake_case.dart`。
- 优先使用 package import，减少深层相对路径。

## 7. 校验门禁（最小必过）
提交前至少通过：
- `dart format`（变更文件）
- `flutter analyze`（变更模块）
- 对应测试（新增或受影响测试）

## 8. 冲突处理
当本文件与项目现状冲突时：
- 优先保持现有可运行架构；
- 在不破坏现网逻辑前提下做增量优化；
- 不做“为了符合模板而改模板外”的破坏性改动。
